# `/office-hours` 技能深度注解

> 对应源文件：[`office-hours/SKILL.md.tmpl`](https://github.com/garrytan/gstack/blob/main/office-hours/SKILL.md.tmpl)
> 这是 gstack 的"YC 产品诊断"技能，也是整个技能链的起点。

---

## 这个技能是什么？

`/office-hours` 模拟 YC（Y Combinator）的 Office Hours：创始人带着想法来，合伙人用尖锐的问题帮你看清现实。

**两种模式：**
- **Startup 模式**：6 个逼问（Forcing Questions），逼你面对需求真相
- **Builder 模式**：设计思维头脑风暴，适合开源/黑客松/学习/副项目

**输出：设计文档（design doc）**，不是代码。

## 6 个逼问（Startup 模式核心）

| # | 问题 | 用意 | 红旗信号 |
|---|------|------|---------|
| Q1 | **需求现实**：谁会因产品消失而难过？ | 区分"有兴趣"和"真需求" | "500人在waitlist" |
| Q2 | **现状**：用户现在怎么解决？ | 真正竞品通常是Excel+Slack | "没有方案"=痛点不够 |
| Q3 | **极致具体**：描述最需要的那个人 | 从品类降维到个人 | "医疗企业"不是人名 |
| Q4 | **最窄楔子**：最小可付费版本？ | 阻止平台思维 | "需要完整平台" |
| Q5 | **观察与意外**：看过用户使用吗？ | 发现真正的产品 | "发了问卷"（会撒谎） |
| Q6 | **未来适配**：3年后更重要还是更不重要？ | 区分搭便车和真论点 | "市场增长20%" |

## 关键设计决策

| 决策 | 原因 |
|------|------|
| Anti-Sycophancy 规则（禁止讨好） | YC 式诊断 = 必须诊断，不能讨好 |
| HARD GATE 不写代码 | 想法验证和实现完全解耦 |
| 两种模式 | 副项目不需要被逼问需求现实 |
| 对抗性自审 | 写完设计文档后自我攻击找漏洞 |
| 设计文档写入 ~/.gstack/ | 持久化，后续评审链可引用 |
