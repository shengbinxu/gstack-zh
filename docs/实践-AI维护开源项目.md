# 用 AI 维护一个开源项目：从创建到持续迭代的完整实践

## 一、这个项目是干什么的

### 背景

[garrytan/gstack](https://github.com/garrytan/gstack) 是 YC 合伙人 Garry Tan 开源的 AI 工程师工具库，包含 40 个工程化"技能"（skills），覆盖代码规划、评审、发布、QA、调试等完整研发流程。每个技能本质上是一个精心设计的 Markdown 文件，Claude 读取后严格按步骤执行，把整个工程流程编码为可复现的 prompt 指令。

这套工具值得深入研究，但它全是英文，且设计原理散落在 2000+ 行的 Markdown 里，直接上手学习成本很高。

### 项目定位

[gstack-zh](https://github.com/shengbinxu/gstack-zh) 是我维护的中文深度注解版本，不是简单翻译，而是**逐段解读设计原理**：每段保留英文原文，配中文翻译，再加一节"为什么这样设计"的分析。

目标读者是想把 gstack 用到自己项目里、或者想基于它构建公司内部工具库的工程师。

### 核心内容

- 38 个技能的逐段中英对照注解（`docs/skills/`）
- 实现原理文档：模板编译管线、Preamble 系统、`~/.gstack/` 状态存储、/ship 完整工作流
- 公司库构建指南：哪些直接复用，哪些必须重新设计
- 完整开发工作流步骤手册

## 二、这个项目是如何运作的

这个项目有一个特点：**上游 gstack 在持续迭代**，每次更新都可能新增技能或修改已有技能的设计，注解需要跟着同步。

人工维护的问题是：检测变更、对比差异、更新注解、提交 PR，这个流程重复且机械。我把它完全自动化了。

### 整体架构

```
上游 gstack 有新 commit
        ↓
GitHub Actions 每日检测（sync-check.yml）
        ↓
自动创建 Issue（列出变更文件、新 SHA）
        ↓
本地定时脚本监测到 Issue（每4小时）
        ↓
调用本地 Claude Code 处理
        ↓
Claude 读上游文件 → 更新注解 → git commit
        ↓
自动创建 PR
        ↓
人工 review → merge → Issue 自动关闭
```

### 第一层：变更检测（GitHub Actions）

`sync-check.yml` 每天 UTC 08:00 运行：

1. 读取 `SYNC.md` 里记录的上次同步 SHA
2. 拉取上游 gstack 最新 HEAD
3. 对比两个 SHA，有新 commit 则计算变更文件列表
4. 自动创建结构化 Issue，包含：变更文件清单、commit 列表、SYNC.md 更新模板

Issue 格式固定，是后续自动化的关键——机器生成、机器消费。一个真实的 Issue 长这样：

```
## gstack upstream update detected

**Last synced SHA:** `7e96fe29`
**Current upstream SHA:** `23000672` (v0.17.0.0)
**Changed files:** 21
**New commits:** 1

## Changed files
```
browse/SKILL.md
design-html/SKILL.md
design-review/SKILL.md
plan-design-review/SKILL.md
...
```

## Checklist
- [ ] Update corresponding annotation docs in `docs/skills/`
- [ ] Update `SYNC.md`:
      upstream-sha: 23000672673224f04a5d0cb8d692356069c95f6a
```

### 第二层：自动修复（本地脚本 + Claude Code）

`auto-sync-issue.sh` 通过 macOS launchd 每 4 小时运行一次，检测到 Issue 后自动处理。

详细机制见下一节。

## 三、auto-sync 的机制

这是整个系统最有意思的部分：一个 200 行的 shell 脚本，把 Claude Code 变成了全自动的文档维护机器人。

### 3.1 脚本的完整流程

```
launchd 每4小时触发
    ↓
git pull main（保持最新）
    ↓
gh issue list --label sync（找开着的 Issue）
    ↓ 没有 → 退出
有 Issue
    ↓
PR 是否已存在？（幂等检查）→ 已存在 → 跳过
    ↓ 没有
clone/更新 upstream gstack 源码
    ↓
创建 feature branch: auto-sync/issue-N
    ↓
claude -p "..." 调用 Claude Code
    ↓ Claude 读文件、写注解、git commit
有新 commit？→ 没有 → 清理分支，退出
    ↓ 有
git push origin auto-sync/issue-N
    ↓
gh pr create（body 写 Closes #N）
    ↓
gh issue comment 贴 PR 链接
```

### 3.2 关键设计：幂等性

脚本会被 launchd 反复触发，必须保证多次执行不产生副作用。

```bash
# 检查这个 Issue 是否已经有 PR 了
EXISTING_PR=$(gh pr list --head "auto-sync/issue-$ISSUE_NUMBER" ...)
if [ -n "$EXISTING_PR" ]; then
  log "PR already exists. Skipping."
  exit 0
fi
```

已有 PR 就跳过，不会重复创建。同理，如果 Claude 跑完没有产生任何变更（说明注解已是最新），清理 feature branch 然后正常退出，不产生空 PR。

### 3.3 关键设计：Issue 内容的安全传递

Issue body 里有 Markdown 格式的反引号、引号、美元符号，直接拼进 shell 字符串会导致转义错误：

```bash
# 危险：特殊字符会破坏 shell
claude -p "处理这个 issue: $ISSUE_BODY"
```

正确做法是写到临时文件，再让 shell 展开文件内容：

```bash
ISSUE_FILE=$(mktemp /tmp/gstack-issue-XXXX.md)
trap 'rm -f "$ISSUE_FILE"' EXIT   # 脚本退出时自动删除
cat > "$ISSUE_FILE" <<EOF
$ISSUE_BODY
EOF

claude -p "
...
$(cat "$ISSUE_FILE")   # 这里安全展开
..."
```

`trap ... EXIT` 保证不管脚本以哪种方式退出（正常、报错、被 kill），临时文件都会被清理。

### 3.4 核心：`claude -p` 非交互模式

```bash
claude -p "完整的任务描述..." \
  --allowedTools "Read,Write,Edit,Bash,Glob,Grep" \
  --output-format text
```

- **`-p`（print 模式）**：非交互，把 prompt 一次性传入，跑完自动退出，适合脚本化调用
- **`--allowedTools`**：明确限制 Claude 能用哪些工具，既是安全边界，也避免 Claude 尝试不相关的操作
- **`--output-format text`**：输出纯文本，可以直接 `tee` 到日志文件

**关键点**：`claude -p` 在本地运行时使用的是 Claude Code 订阅额度，不需要 API key，成本为零。

### 3.5 Prompt 的结构

Prompt 设计是这套系统最核心的工程工作，直接决定输出质量。它分五个部分：

**① 上下文建立**：告诉 Claude 这是什么仓库、目录结构是什么、注解格式规范是什么（同时指向 CLAUDE.md 让 Claude 自己去读）。

**② Issue 内容**：直接把完整 Issue 内容粘进去，包含变更文件列表和新 SHA。

**③ 文件映射规则**：告诉 Claude 如何把上游文件路径映射到注解文件路径：
```
{skill}/SKILL.md → docs/skills/{skill}.md
ARCHITECTURE.md  → docs/architecture.md
scripts/**       → 跳过（源码，不需要注解）
```

**④ 处理规则**：对每个需要更新的注解文件，读上游 → 读当前注解 → 找差异 → 按格式更新。明确禁止删除已有中文分析，除非上游把对应段落删掉了。

**⑤ 约束**：明确告诉 Claude "不要 push，由脚本处理"，避免 Claude 自行决定做额外操作。

### 3.6 PR 自动关闭 Issue

PR 创建时 body 写了 `Closes #N`，这是 GitHub 的内置机制：PR merge 到 main 分支后，对应 Issue 自动关闭。不需要脚本做任何额外操作。

```bash
gh pr create \
  --body "Auto-generated by local Claude Code sync script.
  
  Closes #$ISSUE_NUMBER"
```

### 3.7 launchd vs cron

在 macOS 上选 launchd 而不是 cron，原因：

| | launchd | cron |
|--|---------|------|
| 睡眠唤醒 | 自动补跑错过的任务 | 睡眠期间的任务直接丢失 |
| 日志 | 原生支持 stdout/stderr 重定向 | 需要手动处理 |
| 可靠性 | macOS 原生，系统级管理 | 依赖 crontab 服务 |

plist 配置核心：

```xml
<key>StartInterval</key>
<integer>14400</integer>   <!-- 每4小时，单位秒 -->

<key>StandardOutPath</key>
<string>/Users/xushengbin/.local/logs/gstack-zh-sync.log</string>
```

注册命令：
```bash
launchctl load ~/Library/LaunchAgents/com.gstack-zh.sync.plist
```

## 四、可选方案：在 GitHub Actions 中运行 Claude Code

> 本节介绍一种更强大的方案：把整个自动化流程放到云端，不依赖本地机器。由于需要付费的 Anthropic API key，我目前没有采用，但架构上更完整。

### 核心组件：`anthropics/claude-code-action`

Anthropic 官方提供了专门在 GitHub Actions 中运行 Claude Code 的 Action：

```yaml
- name: Run Claude Code
  uses: anthropics/claude-code-action@beta
  with:
    anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
    github_token: ${{ secrets.GITHUB_TOKEN }}
    direct_prompt: |
      你的任务描述...
```

`direct_prompt` 对应本地方案的 `claude -p`，同样是非交互全自动模式。Claude 在 GitHub 托管的 Ubuntu runner 上运行，可以使用所有文件操作工具和 git 命令。

### 完整 workflow

```yaml
name: Auto Sync Annotations via Claude Code

on:
  issues:
    types: [opened]

jobs:
  auto-sync:
    runs-on: ubuntu-latest
    if: contains(github.event.issue.labels.*.name, 'sync')

    permissions:
      contents: write
      issues: write
      pull-requests: write

    steps:
      - uses: actions/checkout@v4
        with:
          token: ${{ secrets.GITHUB_TOKEN }}

      - name: Create feature branch
        run: |
          git checkout -b "auto-sync/issue-${{ github.event.issue.number }}"
          echo "BRANCH=auto-sync/issue-${{ github.event.issue.number }}" >> "$GITHUB_ENV"

      - name: Clone upstream gstack
        run: git clone --depth 1 https://github.com/garrytan/gstack.git upstream

      - name: Run Claude Code
        uses: anthropics/claude-code-action@beta
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          github_token: ${{ secrets.GITHUB_TOKEN }}
          direct_prompt: |
            # 与本地方案完全相同的 prompt

      - name: Push + Create PR
        run: |
          git push origin "$BRANCH"
          gh pr create --title "..." --body "Closes #${{ github.event.issue.number }}"
```

### 触发时机：Issue 创建即触发

```yaml
on:
  issues:
    types: [opened]
```

Issue 一创建，workflow 立刻触发。不需要等本地机器轮询，响应时间从"最多 4 小时"缩短到"几分钟"。

### 为什么这个方案更强

| 对比项 | 本地方案 | GitHub Actions 方案 |
|--------|---------|-------------------|
| 响应时间 | 最多4小时（轮询间隔） | 几分钟（事件触发） |
| 可用性 | 依赖本地机器开机 | 云端全天候 |
| 维护成本 | 需要管理 launchd、本地环境 | 零运维 |
| 适合场景 | 个人项目、低频更新 | 团队协作、高频更新 |

### 为什么没采用：费用

GitHub Actions 本身免费（public repo），但 `claude-code-action` 调用 Anthropic API，按 token 计费，需要 `ANTHROPIC_API_KEY`。

对于这个 docs 仓库，每次 sync 大约处理 5-20 个文件，消耗 50k-150k tokens，费用约 $0.15-0.50/次。上游 gstack 每月更新几次，月均成本 $1-3。

不贵，但本地方案同样效果、成本为零，所以没有迁移的动力。

如果你的项目：更新频率高、多人协作、或者本地机器经常关机，GitHub Actions 方案更合适。把 API key 存入 GitHub Secrets，把 `.github/workflows/auto-sync.yml` 加到仓库，就能跑起来。

## 五、总结

这个项目展示了一种**把 AI 嵌入开源维护流程**的具体实践：

- **结构化 Issue 是关键**。机器生成的 Issue 格式固定，比人写的更适合机器消费，是整个自动化链路的枢纽。
- **`claude -p` 把 Claude Code 变成命令行工具**。和任何 shell 命令一样可组合，嵌入脚本、定时任务、CI pipeline 都没有障碍。
- **幂等设计避免副作用**。定时脚本会被反复触发，每个步骤都要设计成"多次运行结果一致"。
- **人的精力用在判断上**。机械重复的检测、对比、生成交给自动化，review 质量、决定是否 merge 留给人。

类似的模式可以推广到：依赖库升级后的文档同步、API 变更后的示例代码更新、竞品功能变化后的对比文档维护——凡是"有结构化输入、有固定处理规则、输出是文本"的维护任务，都可以用这套思路自动化。
